## 증상부터

WireGuard를 켰다. ping 잘 나간다. 구글 열린다. 그런데 사내 깃랩에서 클론만 하면 멈춘다. 어떤 사이트는 접속 자체가 한참 걸리다 타임아웃. VPN을 끄면 전부 정상.

이 조합 — "작은 건 되고 큰 것만 조용히 죽는다" — 이면 사실상 MTU 문제다. DNS 조회나 ping은 패킷이 작아서 통과하고, TLS 인증서 체인이나 파일 전송처럼 패킷이 꽉 차는 순간부터 사라지기 시작하는 것.

## 1500에 안 들어간다

이더넷 기본 MTU는 1500바이트다. 여기에 터널을 얹으면 캡슐화 헤더가 그 안에서 자리를 차지한다.

| 구성 | 오버헤드 | 남는 MTU |
|------|---------|---------|
| WireGuard (IPv4 밖) | 60B | 1440 |
| WireGuard (IPv6 밖) | 80B | 1420 |
| PPPoE + WireGuard | 8 + 60 | 1432 |
| IPsec ESP 터널 (worst) | ~73B | 1427 |
| VXLAN | 50B | 1450 |
| GRE | 24B | 1476 |

터널 인터페이스 MTU가 이보다 크게 잡혀 있으면, 그 차이만큼의 패킷은 밖으로 나갈 때 1500을 초과한다. 초과분의 운명은 둘 중 하나 — 조각나거나, 버려지거나.

조합이 위 표에 없으면 [MTU/MSS 계산기](/tools/net/mtu-mss-calculator/)에 스택을 체크하면 된다. 중첩 터널도 계층별로 차감해서 보여준다.

## 왜 에러도 없이 조용히 죽는가

정상이라면 이렇게 굴러간다. 송신 측이 DF(Don't Fragment) 비트를 켜고 큰 패킷을 보낸다 → 중간 라우터가 "너무 크다, 여기 MTU는 1440이다"라는 ICMP 메시지를 돌려준다 → 송신 측이 크기를 줄여 재전송. 이게 경로 MTU 탐색(PMTUD)이다.

문제는 그 ICMP를 방화벽이 먹어버리는 경우다. 보안 장비에서 ICMP를 통째로 막아놓은 네트워크가 생각보다 많다. 그러면 송신 측은 "너무 크다"는 통보를 영영 못 받고, 같은 크기로 계속 재전송하다 타임아웃난다. 이걸 PMTUD 블랙홀이라고 부른다.

에러 메시지가 없는 이유가 이거다. 패킷은 버려졌고, 버려졌다는 통보도 버려졌다.

## 진단은 ping 한 줄

DF 비트를 켜고 페이로드 크기를 지정해서 쏘면 경계가 어딘지 바로 나온다.

```bash
# Linux/macOS — 1472 = 1500 - IP(20) - ICMP(8)
ping -M do -s 1472 8.8.8.8

# Windows
ping -f -l 1472 8.8.8.8
```

1472가 통과하면 그 경로는 1500이 통짜로 뚫려 있는 것. `Frag needed and DF set` 이 뜨면 크기를 줄여가며 이진탐색한다. 1412가 되고 1413이 안 되면 경로 MTU는 1412 + 28 = 1440 — WireGuard 오버헤드 60바이트와 정확히 맞아떨어진다.

터널 안에서 쏘는지 밖에서 쏘는지만 헷갈리지 말 것. 터널 인터페이스를 지나는 경로를 재야 터널의 유효 MTU가 나온다.

## 처방 두 가지

**터널 MTU를 낮춘다.** 가장 정석. WireGuard면 설정에 `MTU = 1420` 한 줄, 리눅스면:

```bash
ip link set dev wg0 mtu 1420
```

1440이 맞는 환경에서도 1420을 쓰는 관례가 있는데, 바깥 경로가 IPv6로 빠질 가능성(오버헤드 80B)까지 덮는 안전값이라 그렇다.

**MSS를 클램핑한다.** 라우터/게이트웨이 입장에서, 지나가는 TCP SYN의 MSS 옵션을 터널에 맞는 값으로 고쳐 쓰는 방법이다. 양 끝이 처음부터 작은 세그먼트로 합의하게 만들기 때문에 ICMP가 막혀 있어도 동작한다.

```bash
iptables -t mangle -A FORWARD -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu
```

`--clamp-mss-to-pmtu` 대신 `--set-mss 1400`처럼 값을 박아도 된다. MSS는 IPv4 기준 MTU − 40이므로, 유효 MTU 1440이면 1400이다.

둘 중 뭘 쓰나 — 내 장비가 터널 양 끝이면 MTU를 낮추는 게 깔끔하다. 남의 클라이언트가 지나가는 게이트웨이 입장이면(사무실 공유기에 WireGuard 사이트-투-사이트를 얹은 경우 같은) MSS 클램핑이 현실적이다. 단 클램핑은 TCP에만 듣는다. UDP 기반(QUIC, VoIP)은 결국 MTU 자체를 맞춰야 한다.

## 자주 밟는 것들

- **PPPoE를 잊는다.** 국내 아파트 광랜은 대부분 1500이지만, PPPoE 회선은 시작점이 1492다. 그 위에 VPN을 얹으면 8바이트를 추가로 빼야 한다.
- **IPv6를 잊는다.** IPv6 헤더는 40바이트라 MSS 여유가 IPv4보다 20바이트 더 필요하다. IPv4만 맞추면 IPv6 트래픽이 같은 증상을 반복한다.
- **MTU를 무작정 1400으로 줄이고 끝낸다.** 동작은 한다. 대신 모든 패킷이 필요 이상으로 잘게 쪼개져 처리량을 몇 퍼센트씩 깎아먹는다. 경계를 재고 딱 맞는 값을 쓰는 게 맞다.

증상 재현과 수치 계산은 [MTU/MSS 계산기](/tools/net/mtu-mss-calculator/)로, 응답이 아예 안 오는 건지 헤더 수준에서 잘리는 건지 확인할 때는 [HTTP 헤더 확인](/tools/net/http-headers-checker/)을 곁들이면 빠르다.
