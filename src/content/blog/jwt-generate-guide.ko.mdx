## 직접 JWT를 만들어야 하는 순간

API를 개발하거나 테스트하다 보면 "유효한 JWT 토큰이 지금 당장 하나 필요한데" 싶은 순간이 옵니다. 보호된 엔드포인트를 Postman으로 찔러봐야 하는데 로그인 플로우를 다 거치기는 번거롭고, 인증 미들웨어가 토큰을 어떻게 검증하는지 확인하려면 직접 만들어 넣어보는 게 가장 빠릅니다.

JWT 디코딩이 토큰을 "열어보는" 일이라면, JWT 생성은 토큰을 "만들어 서명하는" 일입니다. 이 글에서는 JWT의 구조를 다시 짚고, HMAC 서명으로 직접 토큰을 발급하는 과정을 단계별로 설명합니다.

## JWT는 어떻게 만들어지는가

JWT는 점(`.`)으로 구분된 세 부분으로 이루어집니다.

```
Base64URL(Header).Base64URL(Payload).Base64URL(Signature)
```

- **Header**: 서명 알고리즘과 토큰 타입. 예) `{"alg":"HS256","typ":"JWT"}`
- **Payload**: 담고 싶은 클레임(claim). 예) `{"sub":"1234567890","name":"홍길동","iat":1700000000}`
- **Signature**: Header와 Payload를 Secret으로 서명한 값

서명은 다음 식으로 계산됩니다.

```
signature = HMAC-SHA256(
  Base64URL(Header) + "." + Base64URL(Payload),
  secret
)
```

즉 Header와 Payload는 단순히 Base64URL로 **인코딩**될 뿐이고(누구나 디코딩해서 읽을 수 있음), 진짜 보안은 **Secret을 아는 사람만 서명을 재현할 수 있다**는 점에서 나옵니다.

## HMAC 알고리즘: HS256 / HS384 / HS512

HS 계열은 **대칭키** 방식입니다. 토큰을 만들 때와 검증할 때 같은 Secret을 씁니다. 뒤의 숫자는 사용하는 SHA 해시의 비트 수입니다.

| 알고리즘 | 해시 함수 | 서명 길이 | 비고 |
|---------|----------|----------|------|
| HS256 | SHA-256 | 32바이트 | 가장 널리 쓰임. 대부분의 경우 충분 |
| HS384 | SHA-384 | 48바이트 | 더 긴 해시가 필요할 때 |
| HS512 | SHA-512 | 64바이트 | 최고 강도, 토큰 길이도 가장 김 |

선택 기준은 단순합니다. **특별한 요구가 없으면 HS256**으로 충분합니다. Secret만 충분히 길고 무작위(최소 256비트 권장)라면 HS256도 안전합니다.

## 단계별로 JWT 생성하기

[JWT 생성기](/tools/net/jwt-generator/) 도구를 열면 다음 순서로 토큰을 만들 수 있습니다.

1. **알고리즘 선택** — HS256/384/512 중 하나. 선택값이 Header의 `alg`에 자동 반영됩니다.
2. **Header 입력** — 보통 `{"alg":"HS256","typ":"JWT"}` 그대로 둡니다.
3. **Payload 입력** — 필요한 클레임을 JSON으로 작성합니다. 예:
   ```json
   {
     "sub": "1234567890",
     "name": "홍길동",
     "role": "admin",
     "iat": 1700000000
   }
   ```
4. **Secret 입력** — 서버가 검증에 사용할 것과 동일한 비밀 문자열.
5. **생성** — 완성된 토큰을 복사해 `Authorization: Bearer <token>` 헤더에 넣습니다.

만든 토큰이 의도대로 구성됐는지 확인하려면 [JWT 디코더](/tools/net/jwt-decoder/)에 붙여 넣어 Header와 Payload를 다시 열어보면 됩니다. 생성 → 디코딩을 오가며 서명 메커니즘을 눈으로 익힐 수 있습니다.

## 주요 클레임 정리

Payload에 자주 넣는 표준 클레임입니다. 모두 선택이지만 인증 토큰에서는 시간 관련 클레임을 권장합니다.

- `sub` (subject): 토큰 주체(보통 사용자 ID)
- `iat` (issued at): 발급 시각. Unix 타임스탬프(초)
- `exp` (expiration): 만료 시각. 이 시간 이후 토큰은 거부돼야 함
- `nbf` (not before): 이 시각 이전에는 토큰을 사용할 수 없음
- `aud` (audience), `iss` (issuer): 수신자·발급자 식별

만료 시각은 직접 계산해 넣습니다. 예를 들어 "1시간 뒤 만료"는 현재 시각(초) + 3600입니다.

## 자주 묻는 질문 (FAQ)

### Q. 생성한 토큰을 운영 환경에 그대로 써도 되나요?

테스트·디버깅 용도로 권장합니다. 운영 환경의 실제 Secret을 브라우저 도구에 입력하는 것은 피하고, 실제 토큰 발급은 서버의 인증 라이브러리(jose, jsonwebtoken 등)로 수행하세요. 특히 공용 PC에서는 주의해야 합니다.

### Q. Secret이 외부로 전송되나요?

[JWT 생성기](/tools/net/jwt-generator/)는 브라우저의 Web Crypto API로 로컬에서 서명합니다. Secret과 Payload는 어떤 서버로도 전송되지 않습니다.

### Q. RS256(RSA) 토큰도 만들 수 있나요?

이 도구는 HMAC 대칭키(HS256/384/512)만 지원합니다. RS256/ES256 같은 비대칭키 서명은 개인키 관리가 필요해 서버 측 라이브러리를 사용하는 것이 맞습니다.

## 정리

JWT 생성의 핵심은 "Header·Payload는 공개, Secret만 비밀"이라는 점입니다. 서명은 위변조를 막아줄 뿐 Payload를 숨기지 않으므로, 민감 정보를 평문으로 넣지 마세요. 테스트 토큰이 필요할 땐 [JWT 생성기](/tools/net/jwt-generator/)로 즉시 만들고, [JWT 디코더](/tools/net/jwt-decoder/)로 구조를 확인하는 흐름이 가장 빠릅니다.
