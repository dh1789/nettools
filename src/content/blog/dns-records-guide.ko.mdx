## DNS는 인터넷의 전화번호부

브라우저에 `example.com`을 입력하면, 컴퓨터는 먼저 그 이름이 어떤 IP 주소를 가리키는지 알아야 합니다. 이 "이름 → 주소" 변환을 담당하는 것이 DNS(Domain Name System)이고, 그 정보가 담긴 항목 하나하나가 **DNS 레코드**입니다.

도메인을 다루다 보면 "메일이 안 와요", "도메인 인증이 안 돼요", "www는 되는데 루트는 안 돼요" 같은 문제를 만나는데, 대부분 DNS 레코드를 이해하면 풀립니다.

## 자주 쓰는 DNS 레코드

| 레코드 | 역할 | 예시 |
|--------|------|------|
| **A** | 도메인 → IPv4 주소 | `example.com → 93.184.216.34` |
| **AAAA** | 도메인 → IPv6 주소 | `example.com → 2606:2800:...` |
| **CNAME** | 도메인 → 다른 도메인(별칭) | `www → example.com` |
| **MX** | 메일 서버 지정 | `example.com → mail.example.com (우선순위 10)` |
| **TXT** | 임의 텍스트(인증·정책) | SPF, DKIM, 도메인 소유 확인 |
| **NS** | 이 도메인을 관리하는 네임서버 | `ns1.provider.com` |

### A / AAAA — 주소 그 자체

가장 기본. 도메인이 실제로 가리키는 서버 IP를 지정합니다. IPv4는 A, IPv6는 AAAA입니다. 한 도메인에 여러 A 레코드를 두어 간단한 부하 분산을 하기도 합니다.

### CNAME — 별칭

한 도메인을 다른 도메인으로 연결합니다. 흔한 예가 `www.example.com → example.com`입니다. 주의: **루트 도메인(example.com)에는 보통 CNAME을 둘 수 없습니다**(다른 필수 레코드와 충돌). 그래서 루트는 A/AAAA, www는 CNAME으로 두는 구성이 일반적입니다. 일부 DNS 제공자는 이를 우회하는 ALIAS/ANAME을 제공합니다.

### MX — 메일은 어디로

이메일을 받을 서버를 지정합니다. 숫자(우선순위)가 낮을수록 먼저 시도됩니다. "메일이 안 온다"의 상당수는 MX 레코드 누락·오설정입니다.

### TXT — 인증과 정책의 만능 칸

원래 임의 텍스트용이지만, 실무에서는 **도메인 인증**에 가장 많이 쓰입니다. SPF(발신 서버 지정), DKIM(서명 키), 그리고 "이 도메인이 내 것임을 증명하라"는 각종 서비스의 소유 확인 토큰이 모두 TXT로 들어갑니다.

## 더 알아두면 좋은 레코드 (CAA·SRV·PTR·SOA)

기본 5종 외에 실무에서 종종 만나는 것들입니다.

- **CAA**: 어떤 인증 기관(CA)이 이 도메인의 SSL 인증서를 발급할 수 있는지 제한합니다. 잘못 설정하면 발급이 거부되니, [SSL 인증서](/tools/net/ssl-checker/) 발급 문제 시 함께 점검하세요.
- **SRV**: 특정 서비스의 호스트·포트를 알립니다(SIP, XMPP 등). `_service._proto.도메인` 형식.
- **PTR**: IP → 도메인 역방향 조회. 메일 서버는 PTR이 없거나 정방향과 불일치하면 스팸으로 분류되기 쉽습니다.
- **SOA**: 영역(zone)의 권한 정보(주 네임서버, 시리얼, 갱신 주기). 전파·영역 전송 디버깅의 시작점.

## 레코드 확인하기

[DNS 조회](/tools/net/dns-lookup/) 도구에 도메인을 넣으면 A/AAAA/MX/TXT/NS 등 레코드를 한 번에 조회할 수 있습니다. 변경 후 전파가 됐는지, 의도한 값이 맞는지 확인할 때 유용합니다.

명령줄에서는 `dig`가 표준입니다.

```bash
dig example.com A +short
dig example.com MX +short
dig example.com TXT +short

# 특정 리졸버에 직접 질의 (전파/캐시 비교)
dig @8.8.8.8 example.com A +short

# 역방향(PTR) 조회 — 메일 서버 점검에 유용
dig -x 93.184.216.34 +short

# 위임 경로 추적 (루트 → TLD → 권한 서버)
dig example.com +trace
```

## TTL과 전파

각 레코드에는 **TTL(Time To Live)** 이 있어, 리졸버가 결과를 얼마나 캐시할지 정합니다. TTL이 3600(1시간)이면 변경 후 최대 1시간은 옛 값이 남아 있을 수 있습니다. 마이그레이션 전에 TTL을 미리 낮춰두면 전환이 빨라집니다.

## 자주 묻는 질문 (FAQ)

### Q. 변경했는데 왜 바로 반영이 안 되나요?

TTL 때문에 리졸버·브라우저가 옛 값을 캐시하고 있을 수 있습니다. [DNS 조회](/tools/net/dns-lookup/)로 현재 권한 서버가 반환하는 값을 확인하고, TTL만큼 기다리거나 로컬 DNS 캐시를 비웁니다.

### Q. 루트 도메인에 CNAME을 못 쓰는 이유는?

루트에는 SOA·NS 같은 필수 레코드가 있어야 하는데 CNAME은 다른 레코드와 공존할 수 없기 때문입니다. 제공자의 ALIAS/ANAME 또는 루트에 A 레코드를 직접 두는 방식으로 해결합니다.

### Q. SPF/DKIM은 어떤 레코드인가요?

둘 다 TXT 레코드입니다. SPF는 "이 도메인 메일을 보낼 수 있는 서버" 목록, DKIM은 메일 서명 검증용 공개키를 담습니다. 메일 도달률·스팸 분류에 직접 영향을 줍니다.

## 정리

DNS 레코드는 종류만 알면 대부분의 도메인 문제를 스스로 진단할 수 있습니다. 주소는 A/AAAA, 별칭은 CNAME, 메일은 MX, 인증은 TXT — 이 네 가지만 확실히 잡아도 충분합니다. 값 확인이 필요할 땐 [DNS 조회](/tools/net/dns-lookup/)로 실제 응답을 보세요.
