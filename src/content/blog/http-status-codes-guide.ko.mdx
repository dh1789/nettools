## 상태 코드는 "누구 잘못인지"부터 알려준다

HTTP 상태 코드의 첫 자리는 책임 소재의 1차 분류입니다.

| 범주 | 의미 | 책임 |
|------|------|------|
| 1xx | 정보 (처리 계속) | — |
| 2xx | 성공 | — |
| 3xx | 리다이렉트 | 설계 의도 (보통 정상) |
| 4xx | 클라이언트 오류 | **요청한 쪽** |
| 5xx | 서버 오류 | **응답하는 쪽** |

장애 티켓을 받았을 때 "4xx면 호출부·인증·요청 형식부터, 5xx면 서버·업스트림부터" — 이 한 줄이 초동 대응 시간을 줄입니다. 개별 코드 검색은 [HTTP 상태 코드 사전](/tools/net/http-status-dictionary/)이 빠릅니다.

## 헷갈리는 쌍 정리

### 301 vs 302 — SEO가 갈린다

- **301 Moved Permanently**: 영구 이동. 검색엔진이 링크 평가를 새 URL로 이전합니다. 도메인 이전·HTTPS 전환에 사용.
- **302 Found**: 임시 이동. 평가가 이전되지 않습니다. A/B 테스트·점검 페이지처럼 "곧 돌아올" 때만.

실수 패턴: 사이트 이전에 302를 쓰면 검색엔진이 옛 URL을 계속 색인해 SEO 손실이 몇 달 갑니다. 반대로 임시 점검에 301을 쓰면 브라우저·CDN이 영구 캐시해 점검 후에도 옛 경로로 안 돌아옵니다. 301은 브라우저가 강하게 캐시하므로 롤백이 어렵다는 점을 기억하세요.

### 401 vs 403 — "누구세요" vs "너는 안 돼"

- **401 Unauthorized**: 인증 실패 — 자격 증명이 없거나 틀림. 응답에 `WWW-Authenticate` 헤더가 따라옵니다. "로그인하세요."
- **403 Forbidden**: 인증은 됐지만 권한 없음. "누군지 알겠는데, 이 리소스는 안 됩니다."

API 디버깅 순서: 401이면 토큰 만료·헤더 누락(`Authorization: Bearer ...`)부터, 403이면 권한(스코프·역할)과 IP 차단 목록부터 봅니다. 만료 여부는 [JWT 디코더](/tools/net/jwt-decoder/)로 토큰의 `exp`를 열어보면 10초 만에 판정됩니다.

### 502 vs 504 — 업스트림이 "죽었다" vs "느리다"

리버스 프록시(Nginx·ALB·Cloudflare) 뒤에 앱이 있는 구조에서:

- **502 Bad Gateway**: 업스트림이 **연결을 거부하거나 비정상 응답**. 앱 프로세스 다운, 포트 불일치, 컨테이너 재시작 중이 전형적 원인.
- **504 Gateway Timeout**: 업스트림이 **제한 시간 안에 응답을 못 함**. 느린 쿼리, 데드락, 커넥션 풀 고갈이 전형적 원인.

같은 "게이트웨이" 계열이지만 대응이 다릅니다 — 502는 "프로세스 살아있나"(`systemctl status`, 컨테이너 상태), 504는 "무엇이 느린가"(슬로우 쿼리 로그, APM)로 갑니다.

### 404 vs 410 — 없음 vs 지웠음

- **404 Not Found**: 리소스 없음(일시적일 수도).
- **410 Gone**: 의도적으로 영구 삭제됨. 검색엔진이 색인에서 더 빨리 제거하는 경향이 있습니다.

## curl로 재현하는 습관

브라우저는 리다이렉트를 자동 추적해 원인을 가립니다. 상태 코드 디버깅은 curl로:

```bash
curl -I https://example.com/path            # 헤더만 (첫 응답 코드 확인)
curl -sIL https://example.com/path          # 리다이렉트 체인 전부 추적
curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" https://example.com/api
```

리다이렉트 체인에서 어떤 코드로 몇 번 튀는지 보이면 301/302 설정 실수를 바로 잡을 수 있습니다. 응답 헤더 전반 점검은 [HTTP 헤더 확인 도구](/tools/net/http-headers-checker/)로도 가능합니다.

## 로그에서 자주 보는 나머지 코드

| 코드 | 실무 맥락 |
|------|----------|
| 400 | 본문 파싱 실패 — JSON 문법·Content-Type 불일치부터 확인 |
| 405 | 메서드 불허 — 라우트는 있는데 POST/GET 매핑 누락 |
| 408 | 클라이언트가 요청을 너무 느리게 보냄 — 모바일 환경에서 빈발 |
| 429 | 레이트 리밋 — `Retry-After` 헤더 확인 후 백오프 |
| 499 | (Nginx 확장) 클라이언트가 응답 전에 연결 끊음 — 타임아웃 설정 불일치 신호 |
| 503 | 서비스 불가 — 배포 중이거나 헬스체크 실패로 로드밸런서가 제외한 상태 |

429를 만났을 때 무한 재시도를 붙이면 상황이 악화됩니다. `Retry-After`를 존중하고 지수 백오프를 넣는 것이 표준 대응입니다.

## 정리

첫 자리로 책임 소재를 가르고(4xx=클라이언트, 5xx=서버), 헷갈리는 쌍(301/302, 401/403, 502/504)은 위 기준으로 판정하면 로그 해석이 빨라집니다. 재현은 브라우저 대신 `curl -sIL`로. 개별 코드의 의미와 발생 원인은 [HTTP 상태 코드 사전](/tools/net/http-status-dictionary/)에서 한국어로 바로 검색할 수 있습니다.
