## Base64는 "암호화"가 아니다

먼저 가장 흔한 오해부터 짚겠습니다. **Base64는 암호화가 아니라 인코딩입니다.** 누구나 디코딩해서 원본을 그대로 볼 수 있습니다. Base64의 목적은 "숨기기"가 아니라, **바이너리 데이터를 텍스트만 허용하는 통로로 안전하게 통과시키기**입니다.

이메일, URL, JSON, HTTP 헤더처럼 텍스트 기반 매체에 이미지·파일 같은 바이너리를 실어야 할 때 Base64가 등장합니다.

## 어떻게 동작하는가 — 3바이트가 4문자로

Base64의 핵심은 단순합니다.

1. 원본 데이터를 **8비트(1바이트) 단위가 아니라 6비트 단위**로 다시 자릅니다.
2. 6비트는 0~63의 값을 가지므로, 이를 64개의 안전한 문자(`A–Z`, `a–z`, `0–9`, `+`, `/`)로 매핑합니다.
3. 3바이트(24비트) = 6비트 × 4 → 즉 **3바이트가 4문자**로 변환됩니다.

그래서 Base64로 인코딩하면 데이터 크기가 약 **4/3배(약 33% 증가)** 됩니다. 이게 Base64의 비용입니다.

남는 자리는 `=` 패딩으로 채웁니다. 입력 길이가 3의 배수가 아니면 끝에 `=` 또는 `==`가 붙습니다.

```
입력:  "Man" (3바이트)  → "TWFu" (4문자, 패딩 없음)
입력:  "Ma"  (2바이트)  → "TWE=" (1패딩)
입력:  "M"   (1바이트)  → "TQ==" (2패딩)
```

## 직접 비트로 계산해보기 — "Man" → "TWFu"

원리를 한 번 손으로 따라가면 확실히 이해됩니다. 각 문자를 ASCII 코드 → 8비트로 펼친 뒤, 6비트씩 다시 끊습니다.

```
M = 0x4D = 01001101
a = 0x61 = 01100001
n = 0x6E = 01101110

24비트 이어붙이기 : 010011 010110 000101 101110
6비트 값         :   19     22      5     46
Base64 문자      :   T      W       F     u
```

`A`=0, `B`=1 … `Z`=25, `a`=26 … `z`=51, `0`=52 … `9`=61, `+`=62, `/`=63 순서로 매핑하면 `19→T, 22→W, 5→F, 46→u`, 즉 `TWFu`가 됩니다. 입력이 3의 배수가 아니면 마지막 묶음의 빈 비트를 0으로 채우고, 모자라는 4문자 자리를 `=`로 메웁니다.

## 실무에서 Base64가 쓰이는 곳

- **data URI**: 작은 이미지를 별도 요청 없이 HTML/CSS에 직접 박을 때. `data:image/png;base64,iVBOR...`. [이미지 Base64 변환](/tools/net/image-base64-converter/)으로 만들 수 있습니다.
- **이메일 첨부(MIME)**: 메일은 텍스트 프로토콜이라 첨부 파일을 Base64로 인코딩해 전송합니다.
- **JWT**: 토큰의 Header·Payload는 Base64URL로 인코딩됩니다(암호화 아님 — 그래서 민감 정보 금지).
- **HTTP Basic 인증**: `Authorization: Basic <base64(user:pass)>`. 인코딩일 뿐이므로 반드시 HTTPS와 함께 써야 합니다.
- **바이너리를 JSON에 담기**: API에서 작은 바이너리를 문자열 필드로 주고받을 때.

## Base64 vs Base64URL

표준 Base64는 `+`와 `/`를 쓰는데, 이 두 문자는 URL이나 파일명에서 특별한 의미를 가집니다. 그래서 **Base64URL**은 `+ → -`, `/ → _`로 바꾸고 패딩 `=`을 제거합니다. JWT, URL 파라미터 등에서 쓰입니다. 변환 도구를 쓸 때 어느 변형인지 확인하세요.

## 변환해 보기

텍스트나 파일을 [Base64 인코더/디코더](/tools/net/base64/)에 넣으면 즉시 양방향 변환됩니다. 이미지를 data URI로 만들려면 [이미지 Base64 변환](/tools/net/image-base64-converter/)이 더 편합니다.

## 명령줄·코드에서 Base64

브라우저 도구 없이 터미널이나 코드에서 바로 다룰 때 쓰는 한 줄들입니다.

```bash
# Linux / macOS — 인코딩 & 디코딩
echo -n 'Man' | base64            # → TWFu
echo 'TWFu' | base64 -d           # → Man

# 파일 인코딩 (GNU는 76자마다 줄바꿈 → -w0 으로 끔)
base64 -w0 logo.png > logo.b64    # Linux(GNU coreutils)
base64 -i logo.png                # macOS(BSD, 기본 줄바꿈 없음)
```

```python
# Python
import base64
base64.b64encode(b'Man')          # b'TWFu'
base64.urlsafe_b64encode(b'...')  # Base64URL 변형
```

```javascript
// Node.js
Buffer.from('Man').toString('base64')   // 'TWFu'
// 브라우저 — btoa는 Latin1만 처리 (아래 함정 참고)
btoa('Man')                              // 'TWFu'
```

Windows는 `certutil -encode in out`(BEGIN/END 머리말이 붙음) 또는 PowerShell `[Convert]::ToBase64String([IO.File]::ReadAllBytes('file'))`을 씁니다.

## 흔한 함정

- **한글·이모지가 깨진다**: 브라우저 `btoa('한글')`은 Latin1(0–255)만 받아 `InvalidCharacterError`를 던집니다. UTF-8로 먼저 바꿔야 합니다 — `btoa(unescape(encodeURIComponent(s)))` 또는 `TextEncoder`. [Base64 인코더/디코더](/tools/net/base64/)는 UTF-8을 자동 처리하므로 한글도 그대로 됩니다.
- **줄바꿈이 섞여 디코딩 실패**: GNU `base64`는 76자마다 개행을 넣습니다. 토큰·헤더에 그대로 넣으면 깨질 수 있으니 `-w0`으로 끄거나, 디코더가 공백을 무시하는지 확인하세요.
- **패딩 `=` 누락**: Base64URL은 패딩을 빼는 경우가 많은데 엄격한 디코더는 패딩을 요구합니다. 디코딩 전 길이를 4의 배수로 맞춰 `=`를 복원해야 할 수 있습니다.
- **URL에 그대로 넣기**: 표준 Base64의 `+`, `/`, `=`는 URL에서 의미가 충돌합니다. URL·파일명에는 Base64URL을 쓰세요.

## 자주 묻는 질문 (FAQ)

### Q. Base64로 비밀번호를 저장해도 되나요?

절대 안 됩니다. Base64는 디코딩이 자명하므로 보안 효과가 전혀 없습니다. 비밀번호는 bcrypt 같은 해시 함수로 저장해야 합니다.

### Q. 왜 파일 크기가 커지나요?

3바이트를 4문자로 표현하기 때문에 약 33% 증가합니다. 큰 파일을 data URI로 인라인하면 HTML이 비대해지고 캐싱도 안 되므로, 작은 아이콘 정도에만 쓰는 것이 좋습니다.

### Q. data URI는 항상 좋은가요?

작은 자원은 요청 수를 줄여 유리하지만, 큰 이미지는 ① 33% 용량 증가 ② 브라우저 캐시 불가 ③ HTML 비대화 때문에 오히려 손해입니다. 크기와 재사용 빈도를 보고 판단하세요.

### Q. 한글을 Base64로 바꾸면 왜 깨지나요?

대부분 UTF-8 인코딩을 건너뛰어서입니다. 브라우저 `btoa()`는 Latin1만 처리하므로 한글·이모지에서 에러가 납니다. UTF-8 바이트로 변환한 뒤 인코딩하면 됩니다. [Base64 인코더/디코더](/tools/net/base64/)는 이를 자동 처리합니다.

## 정리

Base64는 "바이너리를 텍스트로 안전하게 옮기는 인코딩"이며 암호화가 아닙니다. 3바이트 → 4문자, 약 33% 증가라는 원리만 기억하면 data URI·JWT·MIME 어디서 만나든 헷갈리지 않습니다. 변환이 필요할 땐 [Base64 인코더/디코더](/tools/net/base64/)를 쓰세요.
