## 포트 번호는 왜 있는가

IP 주소는 "어느 컴퓨터"까지만 알려줍니다. 한 서버가 웹, 메일, SSH를 동시에 돌린다면 패킷이 어느 프로세스로 가야 할지 구분할 값이 필요한데, 그것이 **포트 번호**(0~65535, 16비트)입니다. `192.0.2.10:443`처럼 IP와 포트가 합쳐져야 비로소 하나의 통신 종단(소켓)이 됩니다.

## 세 구간 — IANA 포트 체계

IANA는 포트 공간을 세 구간으로 나눠 관리합니다.

| 구간 | 범위 | 용도 |
|------|------|------|
| Well-known | 0–1023 | 표준 서비스(HTTP 80, SSH 22 …). 유닉스에서 바인드에 루트 권한 필요 |
| Registered | 1024–49151 | 벤더/애플리케이션 등록 포트(PostgreSQL 5432, Kafka 9092 …) |
| Dynamic(Ephemeral) | 49152–65535 | 클라이언트가 연결할 때 OS가 임시로 할당 |

주의할 점 하나 — **리눅스의 실제 임시 포트 기본 범위는 IANA 권고와 다릅니다.** `cat /proc/sys/net/ipv4/ip_local_port_range`를 확인하면 대부분 `32768 60999`가 나옵니다. NAT 장비나 방화벽에서 아웃바운드 연결 문제를 볼 때 이 범위를 기준으로 봐야 합니다.

## 실무에서 자주 만나는 포트

전통 서비스와 현대 인프라를 섞어 자주 쓰는 것만 추렸습니다. 전체 검색은 [포트 번호 사전](/tools/net/port-dictionary/)에서 할 수 있습니다.

| 포트 | 프로토콜 | 서비스 |
|------|----------|--------|
| 22 | TCP | SSH |
| 53 | TCP/UDP | DNS — 조회는 [DNS 조회기](/tools/net/dns-lookup/) 참고 |
| 80 / 443 | TCP | HTTP / HTTPS |
| 123 | UDP | NTP |
| 3306 / 5432 | TCP | MySQL / PostgreSQL |
| 6379 | TCP | Redis |
| 8080 / 8443 | TCP | HTTP 대체(프록시·개발) |
| 9092 | TCP | Kafka |
| 2379–2380 | TCP | etcd (K8s 컨트롤 플레인) |
| 6443 | TCP | Kubernetes API 서버 |
| 10250 | TCP | kubelet API |

## 트러블슈팅 복붙 명령

**"이 포트 누가 쓰고 있나?"** — 리눅스:

```bash
ss -tlnp                 # 리스닝 TCP 포트 + 프로세스
ss -tlnp | grep :8080    # 특정 포트만
lsof -i :8080            # 해당 포트를 연 프로세스 상세
```

macOS는 `ss`가 없으므로:

```bash
lsof -iTCP -sTCP:LISTEN -n -P
```

**"원격 포트가 열려 있나?"**:

```bash
nc -zv host.example.com 443          # TCP 연결 시도
nc -zvu host.example.com 53          # UDP (판정은 부정확할 수 있음)
curl -v telnet://host.example.com:443
```

UDP는 연결 개념이 없어 `nc -u`의 "열림" 판정이 부정확합니다. 응답 없음 = 열림일 수도, 필터링일 수도 있습니다.

**"방화벽에서 어떻게 여나?"** — UFW 기준 규칙 생성은 [UFW 방화벽 규칙 생성기](/tools/net/ufw-rules-builder/)로 문법 실수 없이 만들 수 있습니다.

## 보안 관점 — 기본 포트 변경은 얼마나 효과적인가

SSH를 22에서 2222로 바꾸는 고전적 조치는 **자동 스캐너의 소음을 줄이는 효과**는 분명하지만, 보안 경계는 아닙니다. 전체 포트 스캔(nmap -p-)이면 1분 안에 드러나고, 배너로 서비스가 식별됩니다. 포트 변경은 로그 소음 감소용으로 쓰고, 실제 방어는 키 인증·fail2ban·방화벽 소스 제한으로 해야 합니다.

또 하나의 흔한 함정 — **"닫힘(closed)"과 "필터링(filtered)"은 다릅니다.** closed는 호스트가 RST로 응답한 것(호스트는 살아있음), filtered는 방화벽이 패킷을 버린 것입니다. 스캔 결과를 읽을 때 이 구분이 장애 원인 판단을 가릅니다.

## 자주 틀리는 것들

- **1024 미만 바인드 권한** — 컨테이너에서 80 포트 바인드가 실패하면 `CAP_NET_BIND_SERVICE` 케이퍼빌리티 또는 비특권 포트로 우회(8080 + 앞단 프록시)를 검토하세요.
- **TIME_WAIT로 인한 임시 포트 고갈** — 초당 연결이 많은 게이트웨이에서 `ss -s`의 timewait 수치가 수만이면 임시 포트 고갈을 의심. `ip_local_port_range` 확장이나 커넥션 풀링으로 대응합니다.
- **같은 포트, 다른 프로토콜** — TCP 53과 UDP 53은 별개입니다. DNS는 평소 UDP 53을 쓰지만 응답이 512바이트를 넘거나 존 전송(AXFR)이면 TCP 53을 씁니다. 방화벽에서 UDP만 열면 간헐적 조회 실패가 납니다.

## 정리

포트는 "IP 뒤에 붙는 프로세스 주소"입니다. 구간 체계(0–1023 / 1024–49151 / 동적)와 리눅스 실제 임시 범위(32768–60999)를 기억하고, `ss`·`lsof`·`nc` 세 명령이면 대부분의 포트 트러블슈팅이 됩니다. 개별 포트가 궁금할 때는 [포트 번호 사전](/tools/net/port-dictionary/)에서 검색하고, 방화벽 규칙은 [UFW 규칙 생성기](/tools/net/ufw-rules-builder/)로 만들면 빠릅니다.
